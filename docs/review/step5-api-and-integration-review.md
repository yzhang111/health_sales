# 营养辅助工具 Step 5 评审版（API 与集成，简明）

## 这份文档是干嘛的

把系统之间“怎么对接”定清楚：  
前端怎么提交、后端怎么处理、什么时候返回结果、怎么查历史。

---

## 一句话原则

接口简单、状态清晰、错误可解释、操作可追踪。

---

## MVP 接口分组（先够用）

1. 客户与建档
- 新建客户
- 更新客户信息
- 查询客户详情

2. 问题流与评估
- 提交问题流回答
- 触发评估
- 查询评估结果

3. 建议与报告
- 生成建议
- 生成客户报告/销售话术
- 查询报告状态与内容

4. 回访
- 创建回访计划
- 提交回访结果
- 查询回访历史

5. 合规与审计
- 记录转诊事件
- 查询审计日志

---

## 建议的关键接口（简化版）

1. `POST /api/v1/customers`
- 创建客户档案

2. `POST /api/v1/assessments`
- 提交问题流回答并完成评估

3. `POST /api/v1/recommendations`
- 根据评估结果生成建议（含置信度与风险提示）

4. `POST /api/v1/reports`
- 生成客户报告和销售话术卡

5. `POST /api/v1/follow-ups`
- 创建回访计划（默认 2-4 周）

6. `POST /api/v1/compliance-events`
- 记录转诊、免责声明注入、人工覆盖等事件

---

## 返回格式（统一）

成功：

```json
{
  "data": {},
  "meta": {}
}
```

失败：

```json
{
  "error": {
    "code": "validation_error",
    "message": "请求校验失败",
    "details": []
  }
}
```

---

## 状态码（MVP）

1. `200` 查询或更新成功
2. `201` 创建成功
3. `202` 报告生成中（异步）
4. `400` 参数错误
5. `403` 无权限
6. `404` 资源不存在
7. `409` 状态冲突
8. `422` 语义不合法（例如禁忌冲突）

---

## 关键业务约束（接口层必须执行）

1. 命中红线时，拦截个性化销售建议
2. 红线场景下仅允许输出“通用营养参考信息”（如食物与常见维生素方向），且必须附“仅供参考、优先就医”提示
3. 每条建议必须带：触发原因、风险提示、置信度
4. 每份报告必须带免责声明版本号
5. 写操作必须记录审计信息（谁、何时、改了什么）

---

## 最小集成流程（端到端）

1. 销售端提交客户与问题流
2. 系统评估并判断是否红线
3. 非红线则生成建议
4. 生成客户报告 + 销售话术
5. 创建回访任务并进入跟进

---

## 你现在需要拍板的 4 件事

1. 是否同意以上 5 组接口就是 MVP 范围？
2. 是否同意报告生成为异步接口（`202`）？
3. 是否同意统一返回格式与状态码约定？
4. 是否同意“红线拦截个性化建议（仅保留通用参考+就医提示）+ 审计必记”作为接口硬规则？

---

## 评审通过标准（简单版）

满足以下 3 条即通过：

1. 前后端都知道怎么接
2. 错误返回可定位问题
3. 高风险流程不会绕过合规
